# 📝 OB 混淆大数组提取与字典还原战术

## 1. 核心理论概念
- **原理简述**：Javascript Obfuscator (OB 混淆) 的核心保护机制之一是“字符串阵列化”。它会将代码中所有的明文字符串抽取到一个顶部的全局大数组中，然后通过复杂的解密函数（带偏移量计算）在运行时动态获取字符串。
- **应用场景**：应对所有带有标准 OB 混淆特征的风控脚本（如瑞数、各类验证码的核心 JS），找回被隐藏的 API 路径、加密参数名和 DOM 属性。

## 2. 逆向分析切入点
- **特征识别**：代码文件头部通常有一个巨大的包含几百个无意义字符串的数组（如 `const _0x1a2b = ['\x73\x70\x6c', ...]`），紧接着是一个自执行的位移函数，以及一个类似 `_0x3c4d(0x1a)` 的解密分发函数。
- **定位技巧**：在 AST 树中寻找前三个节点，通常对应：1. 数组声明 (`VariableDeclaration`)；2. 数组位移混淆 (`ExpressionStatement`); 3. 解密函数声明 (`FunctionDeclaration`)。

## 3. 核心突破逻辑
- **步骤一：静态剥离**：使用 AST 遍历，将顶部的大数组声明、位移函数、解密函数这三个节点提取出来，并将它们从原 AST 树中 `path.remove()` 删掉。
- **步骤二：沙盒挂载**：将剥离出来的这段纯净解密逻辑，利用 Node.js 的 `vm` 模块（虚拟机沙盒）或 `eval` 运行起来，使其在内存中驻留。
- **步骤三：全局替换**：遍历剩余 AST 树中的所有 `CallExpression`（函数调用）。如果调用的函数名是那个解密函数，则取出传入的参数（如 `0x1a`），传入我们驻留在内存沙盒中的真函数，算出明文字符串。
- **步骤四：节点重塑**：将算出的明文（如 `'split'`）构造成一个新的 `StringLiteral` 节点，替换掉原来的解密函数调用节点。

## 4. 易错点与踩坑记录
- ❌ **错误尝试**：只提取了大数组和解密函数，忘记提取“自执行的位移函数”，导致 AST 算出来的字符串全是乱码或错位的。
- ✅ **正确思路**：必须保证环境的绝对隔离。在沙盒中运行解密函数前，确保清除了浏览器特有的防篡改检测（如正则校验格式），否则解密函数会陷入死循环。
