# 📝 废代码剔除 (Dead Code) 与常量折叠 (Constant Folding)

## 1. 核心理论概念
- **原理简述**：混淆器为了增加分析难度，会插入大量永远不会执行的代码（死区代码）以及将简单的数字/字符串拆散成复杂的表达式（如将 `3` 变成 `0x1 + 0x2`）。常量折叠是在编译期提前计算出结果；废代码剔除则是利用作用域分析删掉那些未被引用的垃圾节点。
- **应用场景**：通常作为反混淆流程的“排头兵”（预处理阶段）和“扫尾员”，用来极大程度降低混淆代码的体积（有时能将代码量缩减 40% 以上），提高后续分析的信噪比。

## 2. 逆向分析切入点
- **特征识别**：满屏幕的 `!+[]` (等于 1)、`!![]` (等于 true)，以及长串的十六进制四则运算。定义了无数个带有业务逻辑名称的函数或变量，但在全局上下文中从未被调用过。
- **定位技巧**：在 AST 中关注 `BinaryExpression` (二元表达式) 和 `UnaryExpression` (一元表达式)。关注 Babel 的 `path.scope.bindings` 对象。

## 3. 核心突破逻辑
- **步骤一：常量折叠 (Constant Folding)**：
  - 拦截所有的运算表达式节点。
  - 使用 Babel 提供的牛逼 API：`path.evaluate()`。它能自动在 AST 层面计算出该节点的最终结果。
  - 检查 `path.evaluate().confident` 是否为 `true`（表示可确信的静态常量）。如果是，直接将原长串表达式替换为算出的 `Literal` 节点（如直接替换为数字 3）。
- **步骤二：死区代码剔除 (Dead Code Elimination)**：
  - 遍历变量和函数声明节点。
  - 获取该节点在当前作用域的引用记录：`const binding = path.scope.getBinding(path.node.id.name);`。
  - 检查 `binding.references`。如果引用次数等于 0，且该节点没有隐藏的副作用（Side Effect），则大胆调用 `path.remove()` 将其彻底斩草除根。

## 4. 易错点与踩坑记录
- ❌ **错误尝试**：盲目依赖引用次数为 0 就删除节点。某些混淆函数虽然没有直接被变量名引用，但它在声明时重写了 `window` 对象（具有副作用），删掉会导致环境失效。
- ✅ **正确思路**：执行 `path.remove()` 前，最好加上 `path.isPure()` 的判断，确保即将被删除的节点不会对全局上下文产生污染或副作用。
