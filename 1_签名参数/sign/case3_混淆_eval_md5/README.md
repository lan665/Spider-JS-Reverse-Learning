# 猿人学第一题 参数逆向分析：动态注入与混淆过渡

## 1. 架构引言：动态代码执行与基础混淆防御

在现代反爬与风控对抗中，防御方深知“前端代码完全透明”的弱点。为了防止逆向工程师通过简单的“全局搜索”找到核心加密入口，开发者通常会采用动态代码注入（Dynamic Code Injection）结合轻量级字典混淆的方式，将真实的加密逻辑掩盖起来。

### 1.1 防护架构分析：动态 eval 注入

目标网站为了保护参数生成逻辑，没有将 MD5 算法直接写在静态的 JS 文件中。相反，它将核心代码进行 Base64 编码后，静默挂载到 `window['b']` 环境对象上。在程序运行到发包的最后一刻，才通过 `eval(atob(...))` 的方式在内存中动态组装并执行。这就好比将武器拆解成零件隐藏，使用时才瞬间组装。

### 1.2 核心痛点：静态搜索失效与反调试干扰

当我们试图逆向寻找签名 `m` 的生成位置时，会面临两个阻碍：

- **静态特征丢失**：无论是搜索 `m=` 还是 `hex_md5`，在源码面板中都一无所获，因为代码是被动态“释放”出来的。
- **无限死循环反调试**：网站埋设了 `setInterval` 配合 `debugger` 的陷阱，一旦打开控制台，页面就会陷入死循环卡死，导致无法正常断点。

### 1.3 核心战术：Ajax Hook 与调用栈溯源

针对这种“隐形”的发包防护，我们的核心战术是**“终点拦截，顺藤摸瓜”**：

1. **越过反调试**：利用浏览器右键的 **Never pause here**（一律不在此处暂停）直接跳过死循环陷阱。
2. **底层发包拦截**：由于全局搜索失效，通过编写 Tampermonkey 脚本进行 Ajax Hook，强行拦截原生的 `$.ajax` 或 `XMLHttpRequest.send`。
3. **调用栈回溯**：触发断点后，不再盲目单步调试，而是通过 **Call Stack**（调用栈）向上穿透，直接空降到动态代码组装的案发现场。

### 1.4 底层原理解析：OB 混淆的初级形态（向 AST 过渡）

在动态注入的代码中，我们会看到大量形如 `J('0x0', ']dQW')` 的函数调用。  
这正是臭名昭著的 **Javascript Obfuscator (OB 混淆)** 的标志性初级形态。它将原本的明文字符串（如 `replace`）抽取到一个大数组中，然后通过 `J` 函数（解密分发器）配合十六进制偏移量去动态读取。这是我们在后续进阶对抗中必须面对的核心敌人。

## 2. 目标接口与抓包分析

- **目标接口**：`/api/match/1`
- **请求方式**：GET
- **加密特征**：URL 参数中包含动态字段 `m`，其值为一段 32 位十六进制字符串与 13 位时间戳的拼接（如 `m=9367ba40fa08bc6c...|1648906681000`）。
- **业务风控**：接口存在时间戳新鲜度校验与 User-Agent 强校验。时间戳精度必须与 JS 原生的 `Date.parse` 抹零逻辑一致，否则会触发拦截。
