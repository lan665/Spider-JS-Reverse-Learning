# 02_加密参数定位体系


> **模块定位：** 本阶段解决 Web 逆向的两大基础先决问题——“加密参数在哪生成的？”以及“它用的是什么标准算法？”。掌握本模块，即可独立完成市面上 60% 未经重度混淆的常规网站逆向。

## 一、 核心知识拆解

本模块的学习笔记分为两大核心版块：

### 🔍 版块一：加密参数定位体系 (Where)
重点攻克数据在浏览器中流转的拦截技术，拒绝盲目全局搜索关键字。
* **发包层定位 (Request Layer)：** 针对 `Headers` 签名、`URL` 参数、`Payload` 密文。
  * 核心战术：Initiator 溯源、Scope 作用域剥离、XHR/Fetch 底层断点。
* **接收层定位 (Response Layer)：** 针对服务器返回的加密 JSON 或高熵密文串。
  * 核心战术：原生 API 劫持（Hook `JSON.parse` 等）、Call Stack 闭包穿透。

### 🔐 版块二：标准密码学与算法还原 (What)
当成功定位到加密现场后，如何一眼识破开发者使用的加密算法，并将其用 Python 还原。
* **散列/哈希 (Hash)：** MD5, SHA-1/256 的特征长度与“魔法常数”。
* **对称加密 (Symmetric)：** AES, DES 的 Key、IV (偏移量)、Mode (模式) 与 Padding (填充规则) 的提取机制。
* **非对称加密 (Asymmetric)：** RSA 公私钥体系在前端的常见实现库（如 JSEncrypt）与长文本分段加密特征。
* **编码机制：** Base64 的魔改识别（自定义码表）与 URL 安全编码。

## 二、 三大模块

## 1. 三大核心实战定位模块
这三大模块分别对应了数据在浏览器中流转的三个关键生命周期。

### 1.1 模块 02-1: 发包层定位 (Request Layer)
核心目标： 解决请求头（Headers）、URL 路径参数（Query String）或请求体（Body）中的动态加密参数（如 U-Sign, X-Bogus, Signature 等）不知从何而来的问题。

必须掌握的技能：

Initiator 链路溯源： 抛弃低效的全局搜索关键字，学会在 DevTools 的 Network 面板中，通过 Initiator 顺藤摸瓜，找到请求发起的根源。

异步栈穿透： 面对 Promise 链导致的 Call Stack 断层，学会利用单步调试（Step Over/Into）配合 Scope 面板，向外层执行上下文寻找真实的业务函数。

XHR / Fetch 原生拦截 (备选战术)： 当框架高度混淆时，学会使用油猴脚本重写 XMLHttpRequest.prototype.send/open 或全局 fetch，在请求发出的最后一刻将其拦截 。

### 1.2 模块 02-2: 接收层定位 (Response Layer)
核心目标： 解决服务器返回的密文数据（如大段高熵的 Base64 字符串）如何在前端页面变成明文的问题。

必须掌握的技能：

解析器劫持 (Parser Hook)： 熟练运用油猴脚本重写全局 JSON.parse，以页面上的可见明文为“诱饵”，在反序列化的瞬间触发 debugger。

闭包穿透与密钥提取： 在拦截到解密现场后，学会在 Call Stack 中上下切换栈帧（Stack Frame），跳出全局 Hook 脚本的作用域，进入 Webpack 打包的私有闭包中，强行提取出 AES/DES 等算法所需的 Key 和 IV。

## 2. 三大底层运行原理
掌握以上操作只是“脚本小子”的水平，只有深刻理解以下底层机制，你才能根据不同的网站防护变种，灵活调整甚至自创定位工具。

### 2.1 前端工程化与拦截器架构 (Axios 洋葱模型)**
必须理解 Vue/React 等现代框架是如何利用 Axios 或自定义 HTTP 客户端的。明白请求拦截器和响应拦截器是如何像“安检机”一样，在数据进出浏览器底层 API 之前，悄悄完成签名添加和密文解密的。

### 2.2 JS 异步事件循环 (Event Loop & Microtask)
必须理解 JavaScript 是单线程的，搞懂 setTimeout (宏任务) 和 Promise (微任务) 的执行顺序。这是解释为什么你在 XHR 层面下断点，往回看 Call Stack 却找不到签名函数的根本原因——因为同步调用栈早就被 Event Loop 清空了。

### 2.3 词法作用域与 Webpack 模块化 (Scope & Closure)
必须理解 JS 的变量可见性规则。明白 Webpack 等打包工具是如何通过 (function(module, exports, require) { ... }) 将各个文件隔离成独立作用域的。理解闭包是如何将密钥锁在内存中不被垃圾回收机制（GC）清理，且无法被全局 window 访问到的。

## 四、 五大核心加密参数位置

根据参数出没的 HTTP 报文位置，主要归纳为以下五大类：

1. **Headers (请求头)**
   * **特征：** 开发者常在此处注入签名（Signature）、时间戳或动态 Token（如 `X-Sign`, `Authorization`, `X-Bogus`），用于防重放与防篡改校验。
2. **Payload / Body (请求体 / 表单数据)**
   * **特征：** 提交密码、敏感业务数据或用户行为轨迹时，常将整个请求体加密为一长串 Base64 或 `data=xxxxx` 格式的密文。
3. **URL Parameters (URL 查询参数)**
   * **特征：** 常用于 GET 请求鉴权，加密参数直接拼接于 URL 之后，如 `?_signature=xxx` 或 `?nonce=xxx`。
4. **Cookie (动态 Cookie)**
   * **特征：** 多见于瑞数、Akamai 等企业级风控系统。表现为高频刷新、长度极长的无意义字符串，由服务端强校验。
5. **Response (响应体数据)**
   * **特征：** 为防中间人劫持或数据爬取，接口返回 AES/DES 等密文数据，需经前端 JS 解密还原后方可进行页面 DOM 渲染。

## 五、 核心定位方法论

### 1. Hook 注入拦截 (主动拦截 / 守株待兔)
* **原理：** 利用 JavaScript 的动态特性，重写（Overwrite）浏览器的原生 API 或目标函数，在执行的必经之路上劫持执行流。
* **特点：** 忽略混淆代码的执行位置，只关注数据最终交由哪个 API 处理。
* **使用时机：** 明确知道数据最终调用的浏览器 API 时。例如：通过 Hook `document.cookie` 的 Setter 拦截动态 Cookie 生成；通过 Hook `JSON.stringify` 拦截对象序列化操作。配合 Tampermonkey 脚本或代理工具（Charles/Fiddler）在页面加载初期注入，触发 `debugger` 后通过调用栈抓取源头。

### 2. 调用堆栈 / Call Stack (回溯追踪 / 顺藤摸瓜)
* **原理：** 代码在断点暂停后，DevTools 会完整记录当前函数的调用链路（栈结构），展示“谁调用了谁”。
* **特点：** 逆向思维，从已知结果倒推执行过程。
* **使用时机：** 成功使代码在发包前（或接收后）暂停时。在确认参数已加密的前提下，通过调用堆栈自顶向下逐层回溯，寻找明文转密文的“分水岭”函数。

### 3. XHR/Fetch 断点 (终点拦截 / 设卡盘查)
* **原理：** 拦截底层网络请求 API，当请求 URL 匹配特定正则或关键词（如 `/api/login`）时，强制暂停 JS 执行。
* **特点：** 精准打击核心网络层，无视前端框架（如 Vue/React）复杂的组件生命周期或模块封装。
* **使用时机：** 明确目标接口 URL 且确认发包时参数已加密。代码通常停在底层的 `XMLHttpRequest.send` 或 `fetch` 处，随后配合 Call Stack 向上跳出底层库，直达业务侧加密现场。

### 4. Initiator / 发起程序 (静态追踪 / 历史档案)
* **原理：** 浏览器网络面板自动记录的请求触发调用链。
* **特点：** 静态追踪，无需提前下断点，抓包后可直接分析。
* **使用时机：** 抓包分析的第一步。用于在充满异步 Promise 的现代前端架构中，跳过 Axios/jQuery 等第三方底层库，快速定位并进入最靠近顶层的业务发包 JS 逻辑中下普通断点。

### 5. 事件 / DOM 断点 (源头跟踪 / 正向溯源)
* **原理：** 监听原生用户交互（如 `click`）或 DOM 节点属性变化的瞬间触发暂停。
* **特点：** 从业务源头出发，进行正向追踪。
* **使用时机：** 加密行为由明确的 UI 动作触发，且由于复杂的异步回调导致 XHR 断点距离加密逻辑过远时。暂停后需配合单步调试（Step into/over）严密监控内存变量变化。

### 6. 本地替换插桩 (Local Overrides / AST 分析)
* **原理：** 利用浏览器 Local Overrides 将线上 JS 映射到本地，强行剥离反调试或插入 `console.log` 甚至自定义 `debugger` 暴露上下文。
* **特点：** 获取代码绝对控制权，无视作用域限制与闭包隐藏。
* **使用时机：** 遭遇无限 Debugger、控制流平坦化高度混淆，或加密函数处于高频循环中（常规断点会导致卡死）时。通过在关键函数的出入口插桩打印 `(明文, 密文)`，快速确认输入输出关系，规避分析冗长的加密算法本身。

### 7. 内存对象溯源 / 拦截器剥离 (Scope & Memory Inspection)
* **原理：** 放弃追踪动态的调用栈执行流，转而利用 DevTools 的 Scope（作用域）面板，直接窥探框架实例（如 Axios、Vue）在内存中的静态数据结构。通过寻找挂载在对象上的真实业务函数，利用 V8 引擎底层暴露的 `[[FunctionLocation]]` 属性，实现跨越时空的源码跳转。
* **特点：** 上帝视角的“降维打击”。无视 Promise 的异步断层，无视繁杂的框架调度链（如 Axios 洋葱模型），直接从“弹药库”里提取开发者编写的真实加解密函数指针。
* **使用时机：** 当成功断住执行流（如触发 Hook 或 XHR 断点），但在 Call Stack 中发现堆栈被 `axios.min.js` 或 `vue.min.js` 等底层框架源码彻底淹没，正向回溯找不到业务层切入点时。此时可扒开当前作用域中的 `this` 对象（例如顺着 `this.interceptors.request.handlers` 寻找），精准定位业务函数出生地。

## 六、 五类加密参数的实战定位策略

为了提升实战效率，将目标参数分为三大战术阵营：

###  第一类：网络发包型 (Headers、Payload、URL Parameters)
**特性：** 最终必须经由 `XMLHttpRequest` 或 `Fetch` 投递至服务端。
* **核心策略：** **XHR/Fetch 断点** 配合 **Initiator 发起程序**。
* **执行路径：** 拦截特定请求 $\rightarrow$ 触发断点 $\rightarrow$ 检查当前参数状态 $\rightarrow$ 依据 Call Stack 向上回溯寻源。
* **辅助策略：** 若针对特定请求头（如 `X-Sign`），可直接 Hook `XMLHttpRequest.prototype.setRequestHeader`。

###  第二类：本地存储型 (Cookie)
**特性：** 既可由服务端 `Set-Cookie` 下发，也可由前端 JS 动态计算写入，通常与发包动作异步解耦。
* **核心策略：** **API Hook 注入**。
* **执行路径：** 编写油猴脚本或代理注入，重写 `document.cookie` 的 Setter 描述符 $\rightarrow$ 拦截写入动作触发 `debugger` $\rightarrow$ 通过 Call Stack 追溯生成算法。

###  第三类：数据解密型 (响应体)
**特性：** 密文由服务端返回，前端 JS 负责在内存中解密以供视图渲染。
* **核心策略：** **关键字全局搜索** 与 **Hook 原生 API**。
* **执行路径：** 优先搜索 `decrypt` 等语义化关键字；或直接 Hook `JSON.parse`，监测解密后的明文对象在何处被反序列化。

## 七、 综合实战：定位方法的优先级与切换时机（排查路径）

在真实复杂的逆向对抗中，没有一种方案是万能的。高效的排查应遵循**“自顶向下，由易到难”**的降级策略。优先使用最直观的工具，遇阻时果断切换到底层拦截手段。

### 第一梯队：优先采用“Initiator (启动器)” —— 快速空降
* **战术定位：** 最理想的路径，试图直接从网络面板空降至业务逻辑层。
* **操作原则：** 抓包后第一眼看 Initiator，尝试点击链路顶层，或寻找包含业务关键词（如 `request`, `api`, `login`）的栈帧，点击进入源码打常规断点。
* **何时果断放弃并切换？**
  * **死链 / 框架迷宫**： 点进去全是 Vue/React 等前端框架的底层调度代码，或者被打包工具封装成了无意义的匿名函数，根本找不到业务逻辑。
  * **异步断层：** 目标网站使用 `setTimeout` 等宏任务彻底切断了执行流，Initiator 栈帧极短，底层完全看不到业务层面发包的源头。
  * **代码泥潭：** 虽能跳入源码，但发现身处数万行的 `chunk.js` 内部，满屏皆是如 `return e.apply(t, n)` 的单字母混淆，无法直观辨认参数的“分水岭”。

### 第二梯队：被迫降级至“XHR / Fetch 断点” —— 守株待兔
* **战术定位：** 当 Initiator 彻底失效（断层或点不进）时，强行从浏览器底层把执行流拉回视野。
* **操作原则：** 利用精确的 URL 路径过滤下断点，在数据包“发出前的一刹那”锁死页面。随后，必须配合 Call Stack（调用堆栈）**由底向上**进行历史回溯，寻找加密函数的入口。
* **何时果断放弃并切换？**
  * **非发包场景：** 目标加密参数是纯本地生成的 Cookie 或其他环境校验，与具体的 XHR 动作不强绑定。
  * **极度混淆 / JSVMP（控制流平坦化）：** 成功断住并回溯堆栈，却发现几十层调用栈全是一模一样的虚拟机解释器循环（如巨大的 `switch-case` 结构），堆栈彻底失去溯源价值。

### 第三梯队：祭出终极杀器“API Hook 注入” —— 半路截杀
* **战术定位：** 降维打击。放弃从起点找终点，直接针对数据的“必经之路”设下埋伏。
* **操作原则：** 编写油猴脚本，针对目标数据的归宿（如 `setRequestHeader` 设请求头、`document.cookie` 写入缓存、`JSON.parse` 数据解密）重写原生 API，触发 `debugger`，从而在离加密逻辑最近的地方捕获上下文。
* **何时果断放弃并切换？**
  * **遭遇强反 Hook 检测：** 目标网站严密检测原生 API 的 `toString()` 甚至原型链状态。一旦发现被篡改，页面直接卡死或下发假数据。此时，只能走向最后的 **AST 脱混淆** 或 **本地插桩 (Local Overrides)** 进行纯静态或沙盒层面的对抗。

### 💡 进阶特种战术：“内存对象剥离”应对框架泥潭

在真实的实战排查中，当你被迫在底层的 `axios.min.js` 或 `Promise.then` 内部断住时，调用堆栈往往会呈现出极具欺骗性的“异步断层”或“框架迷宫”。

此时，**千万不要在底层的调度引擎中死磕单步调试 (Step Into)**。

**破局绝招：**
1. 暂停视线对 Call Stack 的关注，转向右侧的 **Scope (作用域 / 本地变量)** 面板。
2. 检查 `this` 指针或外层闭包变量，寻找框架的配置对象（例如 Axios 的 `this.interceptors.request.handlers` 数组）。
3. 展开这些配置对象，找到状态为 `fulfilled` 的开发者自定义拦截器函数。
4. 展开该函数对象，寻找 V8 引擎底层的 **`[[FunctionLocation]]`** 属性。
5. 点击该属性对应的超链接，浏览器将无视一切异步执行顺序，直接将你“传送”到源码中定义加密 / 解密逻辑的真实案发现场。

> **战术总结：代码的物理排列顺序 $\neq$ 时间执行顺序，而是要掌握内存对象的静态剥离**
